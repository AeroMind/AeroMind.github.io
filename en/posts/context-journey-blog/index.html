0. Introduction
#

How does an Agent “see” the world?

It has no eyes, no ears—just an input window. No matter how complex the outside world is, an Agent can only perceive one thing: Context.

A prompt is context. A tool call’s return value is context. A file’s contents are context. An error message is context.

For an Agent: Everything is context.

This article traces how context evolved from static text into the sole interface through which Agents perceive the world. This evolution is the birth story of Context Engineering.

1. The Origin of Context — In-Context Learning
#

The traditional machine learning paradigm was simple: one task, one model.

Sentiment analysis required one model; machine translation required another. Capabilities were baked into parameters—switching tasks meant retraining.

LLMs broke this paradigm. The GPT series proved something remarkable: a sufficiently large language model can switch tasks through context alone. No retraining needed—just change the prompt.

Traditional models: Task → Training → Specialized model

LLMs: General model + Context → Any task

What does this resemble? The shift from specialized hardware to general-purpose computers.

Early computing was specialized—one machine did one thing. General-purpose computers changed everything: the same machine, with a different program, could do different things.

A program is a computer’s Context.

Even more remarkable is few-shot learning. Give the model a few examples, and it learns a new task:

  • Input: happy → positive
  • Input: sad → negative
  • Input: joyful → ?

The model outputs positive.

No training, no fine-tuning—the examples themselves are the learning. Traditional machine learning required collecting data, engineering features, and training models. LLMs just need a few examples in the context.

This is In-Context Learning: capabilities no longer reside solely in parameters. Context becomes the carrier of capability.

This is the starting point of Context Engineering—we discovered that Context can carry capability, just as programs can carry computational logic.

2. Manually Crafting Context — Prompt Engineering
#

If Context can carry capability, how do we design better Context?

Enter Prompt Engineering.

The simplest approach is role assignment:

You are a senior code review expert…

Then comes structured instructions:

## Task Review the following code
## Requirements 1. Check for security vulnerabilities 2. Check for performance issues 3. Provide improvement suggestions
## Code ...

More advanced is Chain-of-Thought:

Let’s think step by step…

And ReAct, which alternates between reasoning and action:

Thought: I need to first look at the file structure Action: list_files Observation: … Thought: Now I need to…

If we think of an LLM as a program, the Prompt is like its configuration file—defining behavior patterns, input formats, and output specifications. Good configuration makes program behavior predictable and controllable.

What is the essence of Prompt Engineering? Manually crafting every token to maximize the value of a limited context window.

But the limitations are obvious: manual, static, and unscalable.

When your knowledge base has a million documents, you can’t manually stuff relevant content into a prompt.

3. Automatic Context Retrieval — RAG
#

Limited context window, unlimited external knowledge. What do we do?

RAG (Retrieval-Augmented Generation) provides the answer: let relevant information come to you. User asks → Retrieve relevant documents → Inject into Context → Generate response. No need to cram all knowledge into the prompt—just retrieve the most relevant parts when needed.

What does this resemble? A program reading data from a file system instead of hardcoding everything.

The Scarcity of Context
#

Here we introduce a key concept: Context Rot.

As the number of tokens in the context window increases, the model’s ability to accurately recall information degrades. The Transformer architecture requires each token to attend to all other tokens, creating n² relationships. As n grows, attention gets diluted.

This means: Context is a scarce resource with diminishing marginal returns. More isn’t always better—you need to find the smallest possible set of high-signal tokens.

RAG’s Limitations
#

RAG is essentially a fixed-pattern search tool: fixed trigger, fixed process, fixed capability.

Its breakthrough: moving from “manually stuffing Context” to “automatically fetching Context.” But its limitation is also “fixed”—what if the Agent needs to execute code or call an API instead of retrieving documents?

4. Dynamically Generating Context — Tools
#

RAG is a special case of tools. Tools are the generalization of RAG.

RAG: Fixed trigger → Retrieval → Return text Tools: Agent decides → Any tool → Return result

Let’s clarify a concept here: Workflows are systems where LLMs and tools are orchestrated through predefined code paths. Agents are systems where LLMs dynamically direct their own processes and tool usage. The key difference is who makes the decisions.

Tools make Agents possible: not just retrieval but execution, not just fixed triggers but Agent-driven decisions, not just one tool but a toolkit. This resembles executable programs in Unix—cat reads files, grep searches, sh executes commands. Simple, focused, composable.

Tool outputs are injected into Context, becoming the Agent’s new perception. This creates a loop: actions produce new Context, new Context drives the next action. For the first time, Agents can “do things” and “see results.”

MCP: Standardizing Tools
#

As Agents need to connect to more external systems, each system requires separate integration code. MCP (Model Context Protocol) solves this—it’s USB-C for LLMs.

Think of MCP as a universal connector. Just as USB-C provides a universal connector for your phone, laptop, and other devices, MCP provides a universal format for LLMs to connect with external systems. MCP provides a unified protocol, letting Agents connect to all MCP-supporting services the same way.

MCP’s significance goes beyond reducing duplicate work. It makes Tools a discoverable, composable ecosystem. Agents don’t need to know all tools in advance—they can discover and use new tools at runtime.

5. Encapsulating Context Patterns — Skills
#

With tools, Agents can do many things. But problems emerge:

Too many tools, not enough Context. Each tool’s definition, parameters, and return values consume tokens. When an Agent connects to dozens or hundreds of tools, tool definitions alone can eat up significant context window space.

And atomic tools are too low-level. Reading files, writing files, executing commands—these are raw materials for capability, not capability itself.

“Code review” is a capability. It requires:

  • Reading code files
  • Understanding project structure
  • Checking against specific standards
  • Outputting structured review comments

How do we combine these into a reusable capability?

Skill: Discoverable, Loadable Context Packages
#

The essence of a Skill is: an organized folder of instructions, scripts, and resources that Agents can discover and load dynamically to perform better at specific tasks.

This definition has three key points:

  1. Folder structure: Not just a prompt, but an organized collection of resources
  2. Discoverable: The Agent knows which Skills are available
  3. On-demand loading: Only loaded into Context when relevant

The core design principle of Skills is Progressive Disclosure:

Why this design? Because Context is a scarce resource.

If all Skills’ complete contents were stuffed into Context, it would trigger Context Rot. Progressive disclosure lets Agents load only needed information, maintaining high signal-to-noise ratio in Context.

Skill vs Shell Scripts
#

Unix philosophy shines here again: compose small programs with pipes to accomplish complex tasks.

cat access.log | grep "ERROR" | sort | uniq -c | sort -rn | head -10

This command combines 6 small programs to accomplish “find the 10 most common errors.”

Skills and Shell scripts share similarities:

  • Tool → Single command (cat, grep, sort)
  • Skill → Shell script (workflow combining multiple commands)

But Skills are more flexible than Shell scripts:

  • Shell scripts are fixed processes
  • Skills are guidance + resources—Agents can apply them flexibly

Skills can also include executable code. Large language models excel at many tasks, but certain operations are better suited for traditional code execution. For example, sorting a list via token generation is far more expensive than simply running a sorting algorithm. Beyond efficiency concerns, many applications require the deterministic reliability that only code can provide.

From a Context Engineering perspective, Skills are reusable Context patterns—predefined specifications for what information should enter Context, in what order, and how it should be processed.

6. Managing Context — The Challenge of Long-Running Tasks
#

Previous chapters covered Context sources: where to get information.

But there’s another dimension: Context management. When tasks span tens of minutes or even hours, generating tokens far exceeding the context window, what do we do?

Compaction: Compressing History
#

The most direct approach is compaction.

Compaction is the practice of taking a conversation nearing the context window limit, summarizing its contents, and reinitiating a new context window with the summary.

The art of compaction lies in choosing what to keep. Overly aggressive compaction can result in the loss of subtle but critical context; too conservative fails to achieve the goal.

A low-risk compaction strategy is clearing tool call results. Once a tool has been called deep in the message history, why would the agent need to see the raw result again?

Structured Note-taking: Agentic Memory
#

A more elegant approach is letting the Agent take its own notes.

Structured note-taking, or agentic memory, is a technique where the agent regularly writes notes persisted to memory outside of the context window. These notes get pulled back into the context window at later times.

This is like humans taking notes: we don’t remember every detail, but we record key points and consult them when needed.

An interesting example is Claude playing Pokémon. The agent maintains precise tallies across thousands of game steps—tracking objectives, remembering which key achievements it has unlocked, and maintaining strategic notes of combat strategies that help it learn which attacks work best against different opponents. After context resets, the agent reads its own notes and continues multi-hour training sequences or dungeon explorations.

Sub-agent Architectures: Divide and Conquer
#

The third approach is sub-agent architectures.

Rather than one agent attempting to maintain state across an entire project, specialized sub-agents can handle focused tasks with clean context windows. The main agent coordinates with a high-level plan while sub-agents perform deep technical work. Each sub-agent might explore extensively, but returns only a condensed, distilled summary of its work.

A sub-agent might consume tens of thousands of tokens exploring, but returns only 1,000-2,000 tokens of distilled results. This achieves separation of concerns—detailed search context is isolated within sub-agents, while the main agent focuses on synthesizing and analyzing the results.

Which Approach to Choose?
#

  • Compaction: Maintains conversational flow for tasks requiring extensive back-and-forth
  • Note-taking: Excels for iterative development with clear milestones
  • Sub-agent architectures: Handle complex research and analysis where parallel exploration pays dividends

These three approaches can be combined.

Isolated Environments: External Memory Across Sessions
#

When tasks span hours or even days, each new context window is a blank slate. Imagine shift engineers handing off, but each one loses memory of the previous shift—this is the challenge of long-running Agents.

The solution is isolated runtime environments: file systems persist code and documentation, Git records changes that can be rolled back, progress files record work logs. Each session ends by leaving the environment in a “clean state.” The next session reads progress files and git log, resuming work within seconds. Isolated environments are the Agent’s “external memory,” compensating for the context window’s limitations.

The key insight: Even as model capabilities improve, Context management remains a core challenge in scenarios demanding peak performance.

7. The Complete Source of Context — Environment
#

Now let’s step back and see the full picture.

In each previous chapter, we used operating system analogies:

  • Context is like program input
  • Prompt is like configuration files
  • RAG is like file system reads
  • Tools are like executable programs
  • Skills are like applications

This isn’t coincidence. We’re building an operating system for Agents.

  • Environment → Operating system (complete runtime environment)
  • Agent → User/Shell (decision-making entity)
  • Tools → Command-line programs (atomic capabilities)
  • Skills → Applications (packaged capability combinations)
  • Context → stdin/stdout (unified information interface)

This evokes Unix design philosophy: Everything is a file. In Unix, hardware, processes, and network connections are all abstracted as files. Programs interact with the world through a unified file interface.

Unix design philosophy maps perfectly here:

  • Everything is a file → Everything is context
  • Small programs do one thing → Tools do one thing
  • Compose programs with pipes → Compose Tools with Skills
  • Text as universal interface → Context as universal interface
  • Shell as glue → Agent as glue

Key insight: Agents don’t directly touch the Environment—everything is perceived through Context.

Just as Unix programs read input through stdin and output results through stdout, Agents perceive the world and express intent through Context.

Why Coding Agents?
#

Here we address a more fundamental question: Why do we emphasize Coding Agents so much?

Because we believe: Agents are a viable path to AGI. And Coding Agents are the most critical form on this path.

The reason is simple: Any open-ended task can be transformed into computer operations. And code execution is the bridge connecting everything.

Think about it:

  • Write a report → Operate file system + call writing tools
  • Analyze data → Execute Python scripts + generate visualizations
  • Deploy services → Execute command line + configure environments
  • Design products → Call design tool APIs + generate prototypes

Code is the universal glue. An Agent that can write code can theoretically complete any task that can be done with a computer.

In other words: Any task currently done by humans operating computers could potentially be done autonomously by Agents in the future.

The Unique Value of Code Execution
#

Code execution isn’t just about “doing things”—it also solves core Context Engineering challenges:

1. Filter data, reduce tokens

# Without code execution: 10,000 rows all enter Context
tool_call: get_spreadsheet(id="abc123")
→ Returns 10,000 rows, all enter Context

# With code execution: filter in execution environment
data = get_spreadsheet(id="abc123")
pending = [row for row in data if row["status"] == "pending"]
print(f"Found {len(pending)} pending orders")
print(pending[:5])  # Only 5 rows return to Context

2. Protect privacy

Intermediate data can stay in the execution environment without entering the model’s Context. Sensitive information (emails, phone numbers) can be automatically anonymized.

3. Persist state

Agents can write intermediate results to files, continuing work across context resets. They can even save successful code as reusable functions—this is the embryo of Skills.

This is why code execution capability is central to the Environment:

For Coding Agents:

  • Code file contents → Context
  • Terminal execution results → Context
  • Test pass/fail → Context
  • Compilation errors → Context
  • LSP type hints → Context

Environment is the ultimate source of Context. Context is the Agent’s only interface with the world. Code execution is the Agent’s primary way of changing the world.

8. Conclusion: The Full Picture of Context Engineering
#

Looking back at this journey:

Each step answers the same question: How do we give Agents better Context?

  • Prompt Engineering: Manually polish every token
  • RAG: Automatically retrieve relevant information
  • Tools: Dynamically execute to obtain new information
  • Skills: Encapsulate reusable Context patterns
  • Environment: Provide complete information sources

And Context management (Compaction, Structured Note-taking, Sub-agent Architectures) answers another question: How do we maximize Context value within a limited window?

Context Engineering = Sources × Management

All of this points to a bigger picture:

We’re building an operating system for Agents.

Unix proved the vitality of a design philosophy over 50 years:

  • Everything is a file
  • Small programs do one thing well
  • Compose programs with pipes
  • Text as universal interface

The Agent world is recreating this philosophy:

  • Everything is context
  • Tools do one thing well
  • Compose Tools with Skills
  • Context as universal interface

Finally, returning to the opening question:

How does an Agent “see” the world?

The answer is always: Context.

An Agent’s capability ceiling = Context’s quality ceiling

This is Context Engineering.
